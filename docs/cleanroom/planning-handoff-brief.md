## 1. リポジトリと作業場所

- **製品名（npm）:** `typewell-jr-clone`（private monorepo）。
- **構成:** `packages/*`（コアは `@typewell-jr/engine`）、`apps/*`（`apps/web` = Next.js 系 Web UI）。
- **リモート例:** `origin` → GitHub `yokiikoy/twell-clone`（名称は環境依存でよい）。
- **本ブリーフ想定の作業ツリー:** 本体クローンと**兄弟ディレクトリ**の **Git worktree**（例: `…/type-cleanroom`）、ブランチ例: `chore/wordset-cleanroom`。本体と**オブジェクトDBは共有**、**作業ディレクトリとチェックアウトブランチは分離**。

## 2. プロジェクトの背景と目的（非正典だが優先判断の根拠）

一次資料: `docs/re/web-reproduction-roadmap.md`。

**動機（プロダクトが「Web サービスとして補完したい」軸）:**

1. **期間を問わない全期間の実力番付** — 短期イベントではなく、累積序列がコミュニティの共通言語になる土台。
2. **単語系で数十秒を超える打鍵** — 持久・集中・ローマ経路の安定。国語Ｒ相当の **400 確定ストローク**試行が体験の核に近い。
3. **長期不変のワードセットに基づく競技環境** — 何年も変わらない**同一ワードセット**がパターナリティとコミュニティ横断の普遍基準を生む一方、**権利・クリーンルーム**とトレードオフになり得るため、**参照系の持続可能性**は別方針で管理する、と明文化されている。

**VB デコンパイル出力だけで上記を完結できるか（文書上の結論）:**

- **挙動再現・試行体験**には寄与するが、目的全体を「デコンパイラ出力物だけ」で完結させることは**できない**と整理されている。
- 特に **(3) 普遍ワードセット:** デコンパイル由来データは**供出パイプラインの理解・移行・検証**には使えるが、**コミュニティが何年も同じ土俵を使う権利上持続可能な語表**は **クリーンルーム・コーパス・許諾の別レーン**。

**北極星（技術）:** 挙動・データの**可能な限り公式一致**（L4 志向）。ただし **UI はセマンティック一致で可**、**権利は語リスト・ブランド・配布を別途文書化してから**、と制約表がある。

## 3. 「クリーンルーム化」の意図（この文脈での意味）

ロードマップ上の記述に沿うと:

- **目的:** 第三者著作・出荷物由来の文字列を**そのまま再配布・恒久的競技用コーパス**として載せない前提で、**統計・特徴・公開コーパス等から「ほぼ同じ」語セットを再構成**するプロセスを確立したい。
- **トレードオフ:** 本家と**ビット一致の同一語表**は捨てる／別ゲートにする代わりに、**長期同一セット・再配布可能性・説明責任**を取りにいく。
- **エンジニアリング上の住み分け（推奨イメージ）:**
  - **レーン A（解析・回帰・互換検証）:** デコンパイル由来抽出物（現状の `twelljr-*.json` パイプライン）。**リポジトリ内外での再配布はライセンス未確定**として扱う。
  - **レーン B（公開プロダクト・競技用恒久セット）:** クリーンルーム生成または**自著・許諾済みコーパス**のみ。本家語表の**逐語コピー**をしない設計原則を明文化。

※「クリーンルーム」の法学的定義まではこのブリーフは断定しない。**プロダクト要件としての clean rebuild**として計画せよ。

## 4. 現状のワードセット構成（技術）

### 4.1 ソース（本家寄せの解析経路）

- **ローカルのみ（Git 未収録想定）:** `twjrdecomp/*.bas` — VB Decompiler 等の**解析用 IR**。README 方針では **push しない**。
- **抽出の考え方:** `LitStr` を CP932 前提で読み、(日本語表層)(ローマ字)(内部コード) の **triple** としてスキャン（`Kan*` は連続ローマ字で表記揺れ吸収等）。**完全 pcode 再現ではない**（位相ずれ防止・Web 辞書用）。
- **一括再生成:** `npm run extract:wordlists:all -w @typewell-jr/engine` → `packages/engine/scripts/extract-all-decomp-wordlists.ts`。

### 4.2 対象 BAS ステム（11 ファイル = 11 デッキ）

`extract-all-decomp-wordlists.ts` の `STEMS`:

- **基本常用:** `Jou1`, `Jou2`, `Jou3`
- **漢字:** `Kan1`, `Kan2`, `Kan3`
- **カタカナ:** `Kata1`, `Kata2`, `Kata3`
- **慣用句・ことわざ:** `Koto1`, `Koto2`

各ステムから triple 列を取り、次の**二系統**に書き出す:

1. **envelope:** `packages/engine/wordlists-from-decomp/envelope/{Stem}.json`（`schemaVersion`, `source`, `count`, `triples[]` + `index`）
2. **web 配列:** `packages/engine/wordlists-from-decomp/web-array/{Stem}.json` および **`apps/web/public/twelljr-{jou1|…}.json`**（**JSON 配列のみ**: `surface`, `reading`, `code`, `index`）

### 4.3 エンジン上の型（ランタイム辞書）

`packages/engine/src/types.ts`:

- `GameMode`: `"kihon" | "katakana" | "kanji" | "kanyoku"`（**4 つ**）
- `WordEntry`: `surface`, `reading`（ASCII ローマ字）、`typingKana`（**emiel 打鍵ターゲット用ひらがな**）、`mode`, 任意で `sourceDeck?: 1|2|3`（合成出題の重み付け）

**Web の public JSON には `typingKana` が無い**ことが多く、`jouTriplesToWordEntries` が `reading` と `surface` から補完する。

### 4.4 Web UI 側の「デッキ ID」と `GameMode` の対応

`apps/web/src/components/TypingCanvas.tsx` の `DECKS`:

| デッキ ID | JSON URL | `GameMode` | UI グループ |
|-----------|----------|------------|-------------|
| jou1–3 | `/twelljr-jou1.json` … | `kihon` | 基本常用 |
| kan1–3 | `/twelljr-kan*.json` | `kanji` | 漢字 |
| kata1–3 | `/twelljr-kata*.json` | `katakana` | カタカナ |
| koto1–2 | `/twelljr-koto*.json` | `kanyoku` | 慣用 |

**「BAS ファイル名（Jou/Kan/Kata/Koto + 番号）」と「GameMode」は別次元。** 同一 `GameMode` 内で **複数デッキ**があり、`sourceDeck` と **`MERGED_WEIGHTS_*`** で合成試行する。

合成タブの重み（`packages/engine/src/wordPicker.ts`）:

- **kihon（Jou1–3）:** 約 **2:2:1**
- **katakana / kanji（各 3 デッキ）:** **1:1:1**
- **kanyoku（Koto1–2）:** **1:1**

### 4.5 Jou1 表層の「漢字化」補助レイヤ（注意）

`npm run extract:jou:web` パスでは `enrich-jou1-with-kanji-surfaces.mjs` が走り得る。**本家 pcode から RE した公式漢字表記ではない**（`Kan*` 一意照合 + Jisho ヒューリスティック）。**公式 UI 表層との完全一致ゴールデンと混同しない**こと、とパイプライン文書に明記。

### 4.6 試行モデル（Web）

- **400 = 確定ストローク数**（emiel の `finishedStroke.length`）。本家は別基準だったが Web は **Mozc 系曖昧入力優先**のためこの定義を採用、ロードマップに記載。
- ターゲットかな列は **`WordEntry.typingKana` を語間スペースで連結**したものが中核。

### 4.7 その他の語ソース（補助）

- **DetailLog / Detail.txt:** 表層語とキー列からペア推定可能。`docs/spec/behavior-baseline.md`、`scripts/dictionary/parse-detail-log.ts`。**ログの再配布は OSS にそのまま載せない**前提の記述あり。
- **手書きスキーマ例:** `scripts/dictionary/word-list.schema.json`（MVP 用 `WordListFile` 風）。

## 6. クリーンルーム計画で答えてほしい問い（出力してほしいもの）

1. **レーン分離モデル:** 解析用データと公開用データを**どのディレクトリ・パッケージ・ビルドフラグ**で分離するか（単一リポジトリ内での二層化 vs 別リポジトリ）。
2. **「同じ土俵」の定義:** 長期固定セットの**凍結単位**（モード別か、合成か、ストローク分布か、難易度バンドか）。
3. **再構成戦略:** 公開コーパス（例: 辞書・CC 語料・自作）から、**長さ分布・文字種・読み長・語彙帯・モード別件数**などどの**統計指標**で本家セットに「近さ」を定義するか。**逐語一致を目指さない**前提で妥当性を論じること。
4. **検証:** 何を**自動テスト**し、何を**人間監査**（ヘビーユーザ）に残すか。既存の `WordEntry` / `GameMode` / 合成重みとの**互換レイヤ**はどう保つか。
5. **移行:** 既存 `apps/web/public/twelljr-*.json` 利用者・開発者向けの**移行パス**（feature flag、URL、ドキュメント）。
6. **リスク表:** 著作権、商標、「公式と誤認される表現」、競技公平性（語の偏り）、**チート容易性**（ロードマップがプレーンテキスト表層のリスクに言及）を整理。

回答は**箇条書き中心でよいが、各推奨案には根拠とトレードオフ**を必ず添えること。実装タスクに落とす場合は**マイルストーン順**に並べよ。
