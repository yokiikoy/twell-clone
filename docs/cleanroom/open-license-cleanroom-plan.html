クリーンルーム語表（オープンライセンス）— アーキテクチャとリスク（計画）

前提（受領した決定事項）

  • 出力ライセンス（更新）: クリーンルーム語表の配布形態は CC0 に限定しない。ライセンスを明記し、再配布・商用利用可能なオープンライセンスであればよい。CC-BY-SA でも可。プロダクトは本格商用ではなく、コミュニティの投げ銭・サーバー維持費・開発ツール費用程度のマネタイズを想定するため、帰表示・SA 義務（BY-SA の場合）を UI / LICENSE / CORPUS_MANIFEST.json で満たせる設計にする。
  • 入力コーパス: パブリックドメイン、CC0、CC-BY、CC-BY-SA 等、マニフェストに出典・バージョン・条項リンクを残せるものを採用可。BCCWJ 等 利用規約でリスト再配布が禁じられるものは引き続き不可。Wikipedia / Wiktionary 由来を使う場合は、BY（帰属）および SA（改変物の同ライセンス継承）を満たす運用（生成物のライセンス選定、クレジット画面、リポジトリ NOTICE）を アーキテクチャの必須要件とする。
  • 「近さ」: 打鍵プロファイルではなく 単語ベクトル空間での近さ（モード／デッキ別の重心や分布に対するコサイン類似度など）。
  • 品質: SudachiPy の normalized_form 等を用い、異表記同義と 同表記異読を方針どおり複合キーで排除。モード間重複は 要件としてスコープをモード内に限定（本家模倣ならモードごとに独立 Set）。

既存の語ランタイム形状は packages/engine/src/types.ts の WordEntry（surface, reading, typingKana, mode, sourceDeck?）および Web の public/twelljr-*.json（surface, reading, code, index）が正。消費側は apps/web/src/components/TypingCanvas.tsx の DECKS / MERGED_SPECS と packages/engine/src/twelljr/jouSample.ts の jouTriplesToWordEntries。

  • Lane A（現状）: twjrdecomp → npm run extract:wordlists:all → apps/web/public/twelljr-*.json。Git に乗せない／配布しない前提のまま。クリーンルームの入力には使わない。許容するのは オフライン・ローカルでの統計・重心計算のみ。
  • Lane B（新規）: 許可コーパス（PD / CC0 / CC-BY / CC-BY-SA …）→ Sudachi → 単語マスタ（SQLite または JSONL + LICENSE_MANIFEST）→ 埋め込み → スコアリング＋フィルタ → apps/web/public/cleanroom/（新ディレクトリ推奨）または cleanroom-twelljr-*.json など 本家ファイル名と衝突しない命名。出力 JSON の SPDX / CC 表記は、入力の互換集合（例: すべて PD/CC0 なら CC0 出力可、BY-SA 由来が混ざるなら 合成物は BY-SA 等）をビルド時に機械決定するルールをドキュメント化する。
  • 言語スタック: 現リポジトリは TypeScript 中心で Python ファイルは無い。SudachiPy 前提のため packages/cleanroom-pipeline/（Python + pyproject.toml / uv） のような独立パッケージを追加し、npm からは npm run cleanroom:build が薄ラッパで uv run を呼ぶ形が保守しやすい。

「近さ」とストローク制約の統合（妥協案の具体化）

  • スコア: モード／デッキごとに Lane A の語列からベクトルを取り、重心ベクトル（またはクラスタ重心の混合）を Lane B 候補へコサイン類似度でランク付け。
  • ハード制約（必須）: ユーザ受領の「ベクトル＋文字長フィルタ」を、Web 実装と整合するよう ローマ字文字数ではなく typingKana 行に対する emiel 最短ストロークで揃える。既存実装は packages/engine/src/emielStrokeBudget.ts の mozcMinStrokesForHiraganaLine。ビルドパイプライン側では Node 小スクリプトで同関数を呼ぶか、Python 側で同等の近似を避け 単一ソース（TS）でストローク数を列挙してマスタに join するのが安全。
  • 分布: 「平均±N」に加え、パーセンタイル帯（P10–P90）でクリップすると、外れ値語による試行長崩壊をさらに抑えられる（オプション設計として文書化）。

Sudachi による重複排除（アルゴリズムの実装対応）

受領仕様をそのままデータパイプラインに落とす。

  1. トークンごとに (surface, reading_katakana_or_hiragana_policy, normalized_form) を取得。reading は現行 Web と同様 ASCII ローマ字へ正規化（既存 packages/engine/src/twelljr/romajiTypingKana.ts と矛盾しない表記規約を 仕様書で固定）。
  2. モード内ユニーク:
    • 第一段階: normalized の Set で異表記同義を潰す（代表 surface の選び方: 出現頻度最大、または Sudachi の代表表記）。
    • 第二段階: surface の Set で同表記異読を潰す（先勝ち／頻度勝ちを明文化）。
  3. モード間: 上記 Set を モード単位で分離（kihon / kanji / katakana / kanyoku）。GameMode は TypingCanvas.tsx のマッピングに一致させる。

ライセンスの一貫性 複数コーパス（CC0 + CC-BY-SA 等）を混ぜると 出力に課される条件が最も厳しい方へ引き上げられる（典型的には SA）。モデル重みのライセンスも別軸。 LICENSE_MANIFEST で各ソースの条項を固定し、ビルドが合成出力ライセンスを計算して cleanroom/LICENSE または JSON メタに書き出す。モデルはビルド時のみ／リポジトリには重いバイナリをコミットしない。可能なら 同一ライセンス帯のコーパスのみでマスタを切る「ティア」運用。
BY-SA の運用コスト 帰属表示の欠落、改変物のライセンス未継承は ライセンス違反・コミュニティ信頼低下。投げ銭・有料 IDE 程度でも 商用利用の定義はライセンス上は許容されやすいが条項は読む必要がある。 Web に クレジット／データ出典ページ（固定 URL）、リポジトリに NOTICE、ダンプ取得日・ダイジェストを manifest に残す。法解釈はこの計画の正典にしない（必要なら専門家レビュー）。

Web 統合方針（後続フェーズのインタフェース）

  • まず public/cleanroom/*.json を生成し、TypingCanvas に デッキソース切替（環境変数 NEXT_PUBLIC_WORDSET=decomp|cleanroom または開発者向けドロップダウン）を追加するのが衝突が少ない。
  • code 列: 本家互換の内部コードは クリーンルーム語表では意味を持たせない（空・ハッシュID・連番）を決め、jouTriplesToWordEntries が無視できることを確認。

マイルストーン（実装順）

フェーズ1 — 基礎パイプライン（Python）
SudachiPy によるトークナイズ・正規化、許可コーパス取り込み CLI、単語マスタ（surface, reading, normalized, stroke_count 列 optional）と CORPUS_MANIFEST.json / LICENSE_MANIFEST（ソース URL・取得日・ライセンス識別子・条項 URL・帰属テキスト案）。

フェーズ2 — ベクトル
Lane A JSON をローカル読み込み（.gitignore 済み想定）しモード別重心算出。Lane B マスタへベクトル付与・類似度スコア列生成。モデル選定とライセンス文書化。

フェーズ3 — フィルタと出力
類似度ランキング + ストローク分布制約 + 重複排除 → WordEntry 互換 JSON（および既存 web-array 形）を public/cleanroom/ へ。Vitest で「ストローク予算が試行に足る」スモーク（既存 packages/engine/src/wordPicker.test.ts パターンを流用）をクリーンルーム用フィクスチャで追加。
