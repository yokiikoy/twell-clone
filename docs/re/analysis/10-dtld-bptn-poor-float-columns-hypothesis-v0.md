## 4. 解釈のナラティブ仮説（未検証・議論メモ）

「数値だけ見ても分からない」状態でも、**本家の何の画面に紐づくファイルか**から逆算する仮説をメモしておく。以下はいずれも**根拠のない当て推量**ではあるが、後から検証する際の「候補ラベル」として使える。

### 4.1 打鍵間隔（インターバル）系

- **Bptn / Poor** の値が `0.02`〜`0.6` 程度に密集している場合、**秒単位の float**（例: 0.04 ≒ 40ms）や、**正規化された間隔**（0〜1 に押し込めた経過時間）という読み方がある。
- 逆に **Dtld** の先頭付近は `72`, `80` のような**そこそこ大きい整数**も混ざるため、「すべてが秒」ではない可能性が高い。**カウント**（打鍵数・セグメント長）や**ビン集計**（ヒストグラムの各柱の高さ）の方が釣り合うかもしれない。

### 4.2 ミス箇所・弱点スコア系

- ファイル名が `Poor`（弱い）に寄っていることから、**語・キー・モジュール単位の「重み」やミス率」**のテーブルとして並んでいる、という解釈も自然である。
- **Poor** 先頭のゼロ連続は、「そのセッションでは未使用のスロット」か「ミスが無かったビン」など、**疎なベクトル**の埋め草にも見える。

### 4.3 最高記録のリプレイデータか

- **完全なリプレイ**（キー列とタイムスタンプのストリーム）なら、通常は **可変長レコード**や **イベント型のタグ**が混ざることが多い。現状の **Bptn / Poor は float32 の均一列のみ**なので、「生のキーログそのもの」というよりは、**リプレイやセッションから集計した要約系列**（チャート用・内部スコア用）の方が形としては合いやすい。
- **Dtld** はヘッダーに長い識別子文字列があり、その後に float が続く。こちらは「**ある条件セットに対する詳細チャートの点列**」に近いイメージで、**リプレイ本体は別ファイル（例: `Time*.log`）**にあり、Dtld は**同じ試行の派生データ**という関係もありうる（**要相関検証**: 同日・同サフィックスの `Time` ログと件数・時刻を突き合わせる）。

### 4.4 検証するときの最短ルート（人間作業）

1. 本家で**わざとミスが分かる試行**を1回だけ行い、直後に `Dtld` / `Bptn` / `Poor` の更新時刻とファイルサイズを記録する。  
2. 同じ日付の **`Time*.log` のレコード**と並べて、**float 列の長さや先頭数個**が連動するか見る。  
3. **リプレイ再生**があるなら、再生中にだけ値が動くファイルがあるかをフォルダ監視で当たる。
